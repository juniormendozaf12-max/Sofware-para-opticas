1. QUE ES OPTICAL ATELIER
================================================================================

Aplicacion web movil (React) para gestion de un centro optico (Centro Optico
Sicuani). Se conecta a la MISMA base de datos Supabase que el software de
escritorio existente (archivo HTML de 58,000 lineas). Ambos sistemas leen y
escriben los mismos datos en tiempo real.

Funcionalidades:
- Dashboard con estadisticas del dia (ventas, pacientes, stock critico)
- Gestion de ventas con carrito de compras
- Gestion de pacientes (16,000+ registros)
- Prescripciones oftalmologicas (RX)
- Consultas clinicas con diagnostico
- Inventario de productos y matriz de lunas
- Impresion de recetas, tickets y notas de venta
- Integracion con Google Gemini AI (pendiente API key)
- Sincronizacion bidireccional en tiempo real con el software de escritorio

3. BASE DE DATOS — ARQUITECTURA CRITICA
================================================================================

Supabase URL: https://gcrdnnlbetuzrnfqllmn.supabase.co

TABLA UNICA: app_data
  - establecimiento  TEXT      (PK parte 1: 'dos_de_mayo' o 'plaza_de_armas')
  - data_key         TEXT      (PK parte 2: 'clientes', 'ventas', etc.)
  - data             JSONB     (array completo de registros)
  - updated_at       TIMESTAMP (para polling de cambios)

CLAVE PRIMARIA COMPUESTA: (establecimiento, data_key)

Data keys y su contenido:
  - clientes              → Array de 16,000+ pacientes
  - productos             → Array de productos/inventario
  - ventas                → Array de 1,568+ transacciones de venta
  - medidas               → Array de prescripciones RX (nuevas)
  - consultas_clinicas    → Array de consultas clinicas
  - inventario_lunas      → Matriz de inventario de lunas
  - migracion_historial_0 → Chunk 0 de datos migrados (100 pacientes)
  - migracion_historial_1 → Chunk 1 ...
  - ... hasta migracion_historial_163

PATRON DE ESCRITURA:
  Cada escritura hace UPSERT del array COMPLETO. No se modifican elementos
  individuales. El array entero se reemplaza.

PATRON DE LECTURA:
  SELECT data, updated_at FROM app_data
  WHERE establecimiento = 'dos_de_mayo' AND data_key = 'ventas'

PATRON DE MERGE (Desktop):
  El desktop usa _mergeArrayById(local, cloud, 'id') que:
  1. Indexa registros locales por ID
  2. Para cada registro cloud: si es nuevo lo agrega, si existe compara
     fechaModificacion y gana el mas reciente
  3. Resultado es UNION de ambos arrays

5. SERVICES.TS — EL ARCHIVO CENTRAL (DETALLE COMPLETO)
================================================================================

Este archivo contiene TODA la logica de datos. Es el mas importante.

--- MOTOR DE DATOS ---

loadData<T>(key: string): Promise<T>
  Lee datos de Supabase con cache en memoria.
  Cache TTL: clientes=2min, ventas=30s, productos=1min, default=15s.
  Tambien guarda updated_at en _lastUpdated para baseline del polling.

saveData(key: string, items: any): Promise<void>
  Escribe array completo a Supabase via UPSERT.
  Actualiza cache local y _lastUpdated para evitar auto-notificacion.

invalidateCache(key?: string): void
  Limpia cache de un key o de todos.

--- SISTEMA REALTIME (DUAL) ---

onDataChange(key: string, cb: () => void): () => void
  Suscribe un callback a cambios en un data_key.
  Retorna funcion de desuscripcion.
  Auto-inicia WebSocket + Polling en primera suscripcion.
  Auto-para todo cuando no hay suscriptores.

_startRealtime(): void [interno]
  Crea canal WebSocket: supabase.channel('app_data_changes')
  Escucha postgres_changes filtrado por establecimiento.
  Al detectar cambio: actualiza _lastUpdated + llama _notifyKey().

_startPolling(): void [interno]
  setInterval cada 5 segundos.
  Consulta updated_at de todas las keys suscritas.
  Si changed vs _lastUpdated anterior: llama _notifyKey().
  Primera ejecucion solo inicializa baseline (no notifica).

_notifyKey(key: string): void [interno]
  Invalida cache del key.
  Llama todos los callbacks suscritos a ese key.

--- GENERADOR DE IDs ---

uid(prefix: string): string
  Genera: PREFIJO_TIMESTAMP_3RANDOM
  Ejemplo: VEN_1775454190502_462
  Prefijos: CLI_, PROD_, VEN_, RX_, CONS_

--- FUNCIONES DE FECHA (CRITICAS) ---

_localizeDate(d: string): string
  Convierte "2026-04-06" → "2026-04-06T00:00:00"
  NECESARIO porque new Date("2026-04-06") parsea como UTC midnight,
  que en Peru (UTC-5) es el dia ANTERIOR a las 7 PM.
  Con "T00:00:00" JS lo parsea como hora LOCAL.

_toLocalYMD(d: string): string
  Convierte ISO "2026-04-06T15:30:00.000Z" → "2026-04-06" (fecha LOCAL)
  Usa getFullYear/getMonth/getDate (todos locales).
  El desktop REQUIERE este formato exacto en campos fecha/fechaEmision.

--- MAPPERS VENTAS ---

desktopToSale(v: any): Sale
  Convierte formato desktop → formato app.
  Mapeos clave:
    v.fecha → date (via _localizeDate)
    v.estadoPago → status (PAGADO=completed, PENDIENTE=pending, CANCELADO=cancelled)
    v.totalPagar → total
    v.clienteNombre → patientName
    v.vendedor → sellerName
    v.docTipo → documentType (BOLETA=boleta, FACTURA=factura, TICKET=nota)
    v.items[].productoId → items[].productId
    v.items[].importe → items[].subtotal

saleToDesktop(s: Sale, existing?: any): any
  Convierte formato app → formato desktop.
  CAMPOS CRITICOS generados:
    fecha: _toLocalYMD(s.date)           ← YYYY-MM-DD, NO ISO
    fechaEmision: _toLocalYMD(s.date)    ← YYYY-MM-DD, NO ISO
    fechaCreacion: new Date().toISOString()
    fechaModificacion: new Date().toISOString()  ← REQUERIDO para merge
    estadoPago: PAGADO|PENDIENTE|CANCELADO
    estadoEntrega: 'ENTREGADO'
    establecimiento: ESTABLECIMIENTO

--- MAPPERS PACIENTES ---

desktopToPatient(c: any): Patient
  Maneja 2 formatos legacy:
    Nuevo: c.nombre + c.apellidos
    Migrado: c.nombres (campo unico)
  DNI: c.dni || c.documento
  VIP: c._totalVt >= 3 (3+ ventas)

patientToDesktop(p: Patient, existing?: any): any
  Preserva campos existentes con spread.
  Genera nombre/apellidos separados.

--- MAPPERS PRESCRIPCIONES ---

desktopToPrescription(m: any): Prescription
  Campos: lejosOdEsf → od.sph, lejosOdCil → od.cyl, etc.
  DIP: lejosDip (valor completo) o dipOi (valor mitad × 2)

prescriptionToDesktop(rx: Prescription): any
  Inverso. Genera lejosDip (completo) y dipOi (mitad).

--- MAPPERS CONSULTAS ---

desktopToConsultation(c: any): Consultation
  diagMiopia → diagnosis.myopia (boolean)
  Genera diagnosisTags: array de strings legibles

consultationToDesktop(c: Consultation): any
  Inverso. Genera campos diagMiopia, diagAstigmatismo, etc.

--- MIGRACION LEGACY ---

loadMigrationHistory(patientId: string): Promise<{rx, vt} | null>
  Calcula chunk: Math.floor(suffix / 100) donde suffix es el
  numero despues del ultimo '_' en el ID del paciente.
  Carga migracion_historial_N y extrae datos del paciente.
  Retorna {rx: array de prescripciones, vt: array de ventas}.

migratedRxToPrescription(rx, patientId, index): Prescription
  Convierte formato viejo: odE→sph, odC→cyl, odJ→axis

migratedVtToSale(vt, patientId, patientName, index): Sale
  Convierte formato viejo: c→descripcion, t→total, a→pagado

--- OPERACIONES CRUD ---

Pacientes:

8. SISTEMA REALTIME — FLUJO COMPLETO
================================================================================

ESCRITURA (ej: desktop crea una venta):
  1. Desktop hace upsert a Supabase: ventas = [...array_completo]
  2. Supabase actualiza updated_at del row

DETECCION POR WEBSOCKET (instantaneo):
  3. Canal postgres_changes envia payload con data_key='ventas'
  4. _startRealtime() recibe el evento
  5. Actualiza _lastUpdated['ventas'] con el timestamp del payload
  6. Llama _notifyKey('ventas')

DETECCION POR POLLING (fallback cada 5s):
  3b. _startPolling() hace SELECT data_key, updated_at para keys suscritas
  4b. Compara con _lastUpdated → si cambio, llama _notifyKey()

_notifyKey('ventas'):
  7. Borra _cache['ventas'] (fuerza recarga desde Supabase)
  8. Borra _mappedPatients (invalida cache de pacientes mapeados)
  9. Para cada callback suscrito a 'ventas': lo ejecuta

COMPONENTE (ej: Dashboard):
  10. refreshData() se ejecuta (callback silencioso, sin spinner)
  11. Llama fetchDashboardStats() + fetchTodaySales()
  12. Como el cache fue invalidado, hace query fresca a Supabase
  13. Actualiza estado React → UI se actualiza

PREVENCION DE LOOPS:
  - saveData() actualiza _lastUpdated → polling no re-notifica
  - Desktop _isRealtimeSelfEcho: ignora eventos dentro de 5s de su propia escritura
  - loadData() guarda updated_at como baseline → primer poll no dispara falso

12. BUGS CORREGIDOS EN ESTA SESION
================================================================================

BUG 1: Ventas web no aparecen en desktop
  CAUSA: saleToDesktop() guardaba fecha como ISO completo
         ("2026-04-06T15:30:00.000Z") pero el desktop filtra con
         v.fechaEmision !== "2026-04-06" (comparacion exacta string).
  FIX:   Nueva funcion _toLocalYMD() convierte ISO → "YYYY-MM-DD" local.
         saleToDesktop() ahora genera fecha/fechaEmision en formato correcto.

BUG 2: Desktop merge pierde ventas web
  CAUSA: saleToDesktop() no incluia campo fechaModificacion.
         Desktop _mergeArrayById() compara fechaModificacion para resolver
         conflictos. Sin este campo, ventas web siempre perdian.
  FIX:   saleToDesktop() ahora incluye fechaModificacion: new Date().toISOString()

BUG 3: Fechas del desktop se muestran un dia atras en web
  CAUSA: new Date("2026-04-06") parsea como UTC midnight.
         En Peru (UTC-5), eso es 5 de abril 7:00 PM → dia anterior.
  FIX:   Nueva funcion _localizeDate() convierte "2026-04-06" →
         "2026-04-06T00:00:00" que JS parsea como hora LOCAL.
         desktopToSale() ahora usa _localizeDate() para el campo date.

BUG 4: Dashboard muestra "0 ventas del dia"
  CAUSA: fetchDashboardStats() usaba new Date(v.fecha) directo sin
         _localizeDate(). Mismo bug de timezone que Bug 3.
  FIX:   Ahora usa _localizeDate(raw) antes de crear el Date.

BUG 5: Sin sincronizacion realtime en la mayoria de paginas
  CAUSA: Solo PatientProfile tenia suscripciones onDataChange.
         Dashboard, Sales, Patients, Inventory no se actualizaban.
  FIX:   Agregado onDataChange() a todos los componentes con las
         keys relevantes (ventas, productos, clientes, medidas).

BUG 6: Sin fallback si WebSocket falla
  CAUSA: Solo dependia de postgres_changes que requiere configuracion
         del servidor (ALTER PUBLICATION).
  FIX:   Agregado sistema de polling cada 5s que consulta updated_at.
         Funciona sin ninguna configuracion del servidor.

BUG 7: Spinner flash en cada refresh realtime
